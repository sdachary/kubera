Optimizing AiService Structure

Breaking down the 600-line monolithic service into focused components

Current: Monolithic AiService
Monolith
  • AiService.rb (595 lines)
  • Setup Flow Logic
  • OpenAI / Ollama API calls
  • Natural Language Parsing
  • Rule-based Fallbacks
  • Financial Advice Logic
  • Data Formatting (Rupee)
Proposed: Decomposed Architecture
Decomposed
  • Ai::Orchestrator (The coordinator)
  • Ai::SetupFlow (Configuration & detection)
  • Ai::Provider (API integrations)
  • Ai::CommandParser (NL -> Actions)
  • Advice::Base (Financial knowledge base)

Why this is better:

Benefits

  • Maintainability: Smaller files are easier to test and modify.
  • Isolation: Changing the AI provider won't break the setup flow.
  • Reusability: Advice logic can be used in other parts of the app.

Trade-offs

  • More files to manage (but better organized).
  • Small amount of boilerplate for namespacing.
